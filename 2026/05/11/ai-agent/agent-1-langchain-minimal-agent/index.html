知识点拆解

1. Tool Binding — 如何让模型”知道”有哪些工具

bindTools 是 LangChain 提供的核心方法。它做了什么？

  • 把工具列表（包含名称、描述、参数 schema）转换为 OpenAI 兼容的 function calling 格式
  • 注入到每次 LLM 请求的 tools 字段中
  • 模型看到这些工具描述后，自主决定是否调用工具、调用哪个、传什么参数

关键：工具描述的质量直接决定 Agent 的行为。一个模糊的 description 会让模型不知道该何时调用工具。

2. tool() 工厂 — 定义工具的标准方式

metadata 包含三个要素：

字段 作用 影响
name 工具的唯一标识 模型用这个名字来指定调用哪个工具
description 工具的功能描述 模型据此判断当前场景该不该用此工具
schema 参数的 Zod schema 模型按此格式生成调用参数；运行时做类型校验

schema 用的是 Zod——这意味着参数校验和类型推导是自动的。如果模型生成的参数不符合 schema，LangChain 会直接报错而不是传入执行。

3. 多轮推理循环 — Agent 的运行引擎

这是 Agent 最核心的机制：

关键细节：

  • 循环条件：response.tool_calls 存在且非空。只要模型还要求调用工具，循环就继续。模型判断任务完成时会直接返回文本，不包含 tool_calls，循环终止。
  • 消息历史累积：每次模型输出和工具结果都 push 进 messages[]。这意味着模型在每轮推理中都能”看到”完整的对话历史和工具调用链路。
  • 并行工具执行：Promise.all 保证多个工具调用同时执行，不会串行等待。
  • ToolMessage 的角色：工具结果必须用 ToolMessage 包装，并通过 tool_call_id 与原始调用请求关联——这是 OpenAI function calling 协议要求的，不然模型不知道哪个结果对应哪个调用。

4. SystemMessage — Agent 的行为约束

SystemMessage 在 LangChain 中等同于 OpenAI 的 role: "system"。它可以：

  • 定义 Agent 的角色和职责
  • 描述使用工具的流程和优先级
  • 设置行为约束（什么该做、什么不该做）

这个例子里的 SystemMessage 很简略，但在更复杂的 Agent 中，system prompt 往往是几百行的结构化文档——这就是上一篇文章讲的 Skill 的载体。

5. 为什么选 temperature: 0

Agent 场景中对工具调用的一致性要求很高——你不想让它 10 次里有一次忘了调工具。所以 Agent 的默认 temperature 通常是 0 或接近 0 的小数。

整体架构图

关键要点

  1. bindTools 是模型与工具的桥梁 — 把工具元数据注入 LLM 请求，让模型”知道”有什么工具
  2. 多轮循环是 Agent 的灵魂 — 模型调工具 → 执行 → 回传结果 → 模型再推理，循环直到完成
  3. ToolMessage 不可省略 — 它是协议层面的要求，关联调用与结果
  4. 工具描述 = Agent 的说明书 — 描述写得越好，模型调用工具的时机越准
  5. temperature: 0 是 Agent 的标配 — 工具调用的确定性比创造性重要

    Go|Map底层精简

    Fang 2024/9/7 后端Go知识点

    # map

    # 底层原理

    Go中的map是一个指针，占用8个字节，指向hmap结构体。

    源码包中src/runtime/map.go定义了hmap的数据结构：

    hmap包含若干个结构为bmap的数组。

    bmap底层都采用链表结构，bmap通常叫bucket。

    一个桶最多8个位置，key被放入一个桶是因为经过hash运算，低B位是通通的，hash的高八位决定key放到桶的哪个位置。

    bmap结构静态，编译后会拓展。key value以key和value分别放一起的策略，更节省内存空间。

    tophash快速定位key的位置。用key的hash高八位作为tophash的值，tophash不仅能存hash高八位，还能存其他状态。

    当map的key和value都不是指针，bmap不包含指针，那么gc不用扫描bmap。

    bmap指向溢出桶的字段overflow是uintptr类型，为了防止这些overflow桶被gc掉，所以需要mapextra.overflow将它保存起来。如果bmap的overflow是bmap类型，那么gc扫描的是一个个拉链表，效率明显不如直接扫描一段内存。

    # 扩容机制

    # 负载因子

    装载因子越大，填入的元素越多，空间利用率就越高，但发生哈希冲突的几率就变大。反之，装载因子越小，填入的元素越少，冲突发生的几率减小，但空间浪费也会变得更多，而且还会提高扩容操作的次数。

    # 扩容时机

    在向map插入新键时，会进行条件检测，符合以下两个条件之一时触发扩容：

    1. 条件1：超过负载

      • 当map中的元素个数大于6.5 * 桶个数时，触发扩容。这意味着当map的负载因子过高时，可能会影响查找和插入的效率。

    2. 条件2：溢出桶太多

      • 当桶总数小于2^15时，如果溢出桶的数量大于或等于桶的总数，则认为溢出桶过多。
      • 当桶总数大于或等于2^15时，如果溢出桶的数量大于或等于2^15，也认为溢出桶太多。
      • 这个条件是对条件1的补充，旨在解决在负载因子较小的情况下，桶的使用率低、查找和插入效率低的问题。

    # 扩容机制

    1. 双倍扩容

      • 针对条件1，创建一个新的buckets数组，大小为原来的两倍，并将旧的buckets数据搬迁到新的buckets中。

    2. 等量扩容

      • 针对条件2，不扩大容量，保持buckets数量不变，重新排列键值对，以提高桶的利用率和存取效率。此过程类似于双倍扩容，但不改变桶的数量。

    # 扩容函数

    • hashGrow()函数
      • 该函数负责分配新的buckets并将旧的buckets挂到oldbuckets字段上。它并不执行实际的搬迁操作。
    • growMork()函数
      • 该函数负责实际的搬迁操作，将原有的键值对重新分配到新的内存地址。
    • 调用时机
      • 在mapassign和mapdelete函数中调用growMork()，即在插入、修改或删除键时，都会尝试进行搬迁操作。首先检查oldbuckets是否已完成搬迁。

    # 渐进式搬迁

    • Go语言的map扩容采用渐进式的方式，避免一次性搬迁所有键值对。每次最多只搬迁2个桶，以减少延迟，特别是在存储大量键值对时，这种方式可以有效降低性能影响。

    Go语言的map扩容机制通过负载因子和溢出桶的监测，确保在高效使用内存的同时，保持良好的查找和插入性能。通过双倍扩容和等量扩容的策略，以及渐进式搬迁的方式，优化了map的性能和响应速度。

    2. 遍历过程中可能出现key搬迁的情况，又是无序的了

    并发读写出现panic

    2. sync.map

    # 遍历

    写保护监测，计算hash，对应桶，是否扩容，是等廓然碗就bucket找新的key，否直接找key，返回key对应指针或空指针

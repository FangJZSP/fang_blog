## 一、什么是 Skill

如果你用过 AI 编程工具，一定有过这种体验：每次做代码审查都要把同一套标准重新说一遍，每次发版都要把发布流程从头交代一次。这些"重复的上下文"就是 Skill 要解决的问题。

**Skill 本质上是一套可复用的 Agent 行为配置。** 它把某个领域的知识、工作流、约束规则打包成一个"能力单元"，Agent 加载后就能在该领域像专家一样工作。

和普通 prompt 的区别：

| 维度 | 普通 Prompt | Skill |
|------|------------|-------|
| 生命周期 | 一次性，用完即弃 | 持久化，可复用 |
| 结构 | 自然语言指令 | 系统提示词 + 工具权限 + 触发条件 |
| 深度 | 单层指令 | 多步工作流，含分支与边界 |
| 可组合性 | 无 | 多个 Skill 可协作 |

一个直观类比：如果你的 AI Agent 是一部智能手机，Skill 就是 App。裸机能做基本的事，但装上 App 才有专业能力。你不需要每次打开相机都告诉手机怎么调焦——App 已经封装好了。

## 二、Agent 能力分层：Skill 在哪个位置

要理解 Skill 的定位，需要先看清 Agent 的能力分层模型：

```
L4  多 Agent 编排    ← 多个 Skill 各司其职，协作完成复杂任务
     ┌─────────┐
L3   Skill 层       ← 领域知识 + 工作流 + 约束规则（本文主角）
     ┌─────────┐
L2   工具增强层     ← 模型 + 工具调用（读文件、执行命令、搜索）
     ┌─────────┐
L1   裸模型层       ← 单次问答，无工具，无状态
```

- **L1 裸模型**：你问它答。能写代码片段，但没有上下文、没有工具、没有记忆。像一个被困在聊天框里的聪明大脑。
- **L2 工具增强**：模型获得了手和脚——读写文件、执行 Shell、调用 API。这是当前 AI 编程工具的基础形态。
- **L3 Skill 层**：在 L2 之上叠加**领域知识 + 工作流 + 约束规则**。它不是给模型更多的工具，而是教会它"在特定场景下如何正确地使用工具"。Skill 是**纵向能力深化**，而非横向功能堆叠。
- **L4 多 Agent 编排**：多个 Skill 各自激活，像团队里的不同角色——一个人管代码审查，一个人管部署，一个人管文档。

关键洞察：**Skill = Agent 的能力切片。** 一个通用 Agent 加载了 OpenSpec Skill = 一个结构化管理变更的专用 Agent。加载了 Code Review Skill = 一个代码审查专家。同一个 Agent，不同的 Skill 组合，就是不同的专业角色。

## 三、Skill 是怎么创作的——以 OpenSpec 为例

理解 Skill 最好的方式是看一个真实案例。OpenSpec 是 Anthropic 团队发布的一个 Skill，用于结构化地管理项目变更。它的工作流是 **propose → apply → archive**。

### 3.1 第一步：识别重复性工作流

任何 Skill 的起点都是一个被反复执行的任务：

> 每次要改项目，先想清楚要做什么，再实现，最后归档——这个过程每个人都在做，但每次都从头开始组织。

创作 Skill 的第一问永远是：**什么流程在我团队中重复了 10 次以上？** 答案就是这个 Skill 的骨架。

### 3.2 第二步：拆解为结构化制品

把工作流拆成可交付的中间产物。OpenSpec 定义了四个制品：

```
propose 阶段：
  ├── proposal.md     ← 为什么要做，范围是什么
  ├── design.md       ← 技术方案，决策与权衡
  ├── specs/*/spec.md ← 可测试的需求规格
  └── tasks.md        ← 按依赖排序的任务清单

apply 阶段：
  └── 逐项执行 tasks.md，勾选 [x]

archive 阶段：
  └── 变更归档 + delta specs 合并到主规格
```

拆解制品的核心原则：**每一步产出都要有明确的格式和用途，让 Agent 知道"做到什么程度算完成"。**

### 3.3 第三步：编写系统提示词

这是 Skill 的灵魂。系统提示词不是一句指令，而是一个**角色定义文件**，通常包含：

```
┌──────────────────────────────┐
│ 1. 角色定义                   │
│    "你是一个结构化变更管理专家"   │
│                              │
│ 2. 工作流描述                  │
│    propose → apply → archive  │
│    每个阶段的输入输出            │
│                              │
│ 3. 制品规范                   │
│    proposal.md 必须包含哪些章节  │
│    tasks.md 的任务格式         │
│    spec.md 的 WHEN/THEN 模板   │
│                              │
│ 4. 约束规则                   │
│    不能跳过 propose 直接 apply  │
│    archive 前必须所有任务完成    │
│    变更范围不能超出 proposal     │
│                              │
│ 5. 边界与错误处理               │
│    遇到冲突怎么办               │
│    缺少信息时向用户询问          │
└──────────────────────────────┘
```

### 3.4 第四步：绑定工具与触发条件

一个 Skill 的三元组结构：

```yaml
Skill:
  prompt:   <系统提示词>     # Agent 加载后"变身为谁"
  tools:    [Read, Write, Edit, Bash, ...]  # 允许使用哪些工具
  trigger:  "/openspec-*"   # 何时激活（用户输入匹配规则）
```

工具权限遵循**最小权限原则**：只给 Skill 必要的工具。OpenSpec 的 propose 阶段只需要 Read 和 Write 工具，不需要 Bash 执行权限——不给多余的，就是安全。

### 3.5 第五步：迭代打磨

真实环境中会遇到的边界情况：

- 用户中断了 apply，回来继续时怎么恢复上下文？
- 两个 change 同时修改同一个文件怎么处理？
- tasks.md 里的任务粒度太粗或太细怎么办？

这些不是设计出来的，是用出来的。好的 Skill 是在几十次真实使用中磨出来的。

## 四、实战：构建一个类 OpenSpec 的 Skill

讲完理论，我们来动手构建一个。假设我们要做一个 **"代码审查 Skill"**，流程是 `review → report → track`。

### 4.1 定义工作流与制品

```
/review    → 扫描当前分支的改动，生成 review.md（问题清单）
/report    → 将 review.md 渲染为结构化报告
/track     → 跟踪每个问题的修复状态
```

### 4.2 编写系统提示词

```
你是代码审查专家，职责是审查 git diff 并生成结构化审查报告。

工作流：
1. review：执行 git diff，分析改动，按安全/性能/可维护性分类
   记录问题
2. report：将问题清单渲染为 markdown 报告，每个问题含严重
   级别、文件位置、修复建议
3. track：对比当前 diff 与 review.md，标记已修复的问题

约束：
- 每个问题必须引用具体文件和行号
- 严重级别分 blocker/warning/nit 三级
- 不要审查测试文件（路径匹配 *_test.*）
- 如果是重构 PR，降低可维护性类问题的权重
```

### 4.3 绑定工具

```yaml
Skill: code-review
prompt: <上述系统提示词>
tools: [Bash(git diff, git log), Read, Write]
trigger: "/review"
```

一个 Skill 就这样诞生了。虽然简单，但它已经有了**可复用的结构**。

### 4.4 创作 Skill 的核心心法

| 原则 | 含义 |
|------|------|
| **流程驱动** | Skill 描述的是"怎么做"，不是"做什么" |
| **制品导向** | 每一步有明确输出格式，Agent 才知道何时完成 |
| **约束优先** | 约束比自由更重要——约束让 Agent 行为可预测 |
| **最小权限** | 只给必要的工具，减少意外副作用 |
| **边界明确** | 明确告诉 Agent "什么时候停，什么时候问" |

## 五、Skill 的运行时原理

从用户输入 `/review` 到拿到审查报告，背后发生了什么？

### 5.1 整体流程

```
用户输入 "/review"
      │
      ▼
┌─────────────┐
│ 1. 匹配触发   │  系统检测到 "/review" 匹配 code-review Skill
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 2. 上下文注入 │  Skill 的 prompt 被注入到 system prompt 中
│              │  此时 Agent "变身为"代码审查专家
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 3. 多步推理   │  Agent 按 prompt 定义的工作流逐步执行
│              │  第一步：git diff → 第二步：分析 → 第三步：写入 review.md
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 4. 工具编排   │  每步推理产生 tool_call：
│              │  Bash("git diff main...HEAD")
│              │  Read("src/auth.py")
│              │  Write("review.md", report_content)
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 5. 护栏检查   │  权限校验：工具调用是否在允许列表中？
│              │  输出校验：生成内容是否符合制品规范？
└─────────────┘
```

### 5.2 上下文注入机制

这是 Skill 最核心的技术点。System prompt 是分层管理的：

```
┌──────────────────────────────┐
│ 基础层：通用的安全规则、行为准则    │  ← 始终存在
├──────────────────────────────┤
│ 工具层：可用工具列表与调用规范     │  ← 始终存在
├──────────────────────────────┤
│ Skill 层：当前激活的 Skill prompt │  ← 动态挂载/卸载
├──────────────────────────────┤
│ 会话层：当前对话历史与文件上下文    │  ← 持续累积
└──────────────────────────────┘
```

Skill 的注入不是简单的字符串拼接——它需要和基础层、工具层、会话层协同工作。一个关键设计是：**Skill prompt 的优先级高于基础层**。如果 Skill 说"你是代码审查专家"，而基础层说"你是一个通用助手"，Skill 的定义胜出。这就是"变身"的机制。

### 5.3 工具调用编排

Skill 不"执行"工具，Skill 引导 LLM 做**工具选择与多步推理**。关键在于 prompt 中的工作流描述足够清晰，让模型在面对具体场景时能做出正确的工具调用序列决策。

例如 OpenSpec 的 apply 阶段，Agent 面对 `tasks.md` 时的推理链：

```
读 tasks.md → 找到第一个 [ ] 任务 → 理解任务描述 → 决定调用
哪些工具 → 执行 → 标记 [x] → 读下一个任务
```

这不是硬编码的状态机，而是 prompt 引导下 LLM 的涌现行为。好的 Skill prompt 让这种涌现行为**可控、可预测**。

### 5.4 控制平面：护栏与安全

```
用户输入 → [权限检查] → Skill 匹配 → [工具白名单校验]
    → Agent 执行 → [输出校验] → 返回用户

每一层都是护栏：
- 权限检查：用户是否授权了这个 Skill？
- 工具白名单：Skill 只能调用声明过的工具
- 输出校验：生成的文件是否符合预期格式？
```

这就是为什么 Skill 比裸 prompt 更安全——**它在执行路径上设置了多层检查点**。

## 六、生态价值与趋势

### 6.1 从"写 prompt"到"装配 Agent"

AI 编程工具的演进有一个清晰的轨迹：

```
2023: "帮我写一个排序函数"              ← 单次问答
2024: "帮我重构这个模块"               ← 多步工具调用
2025: "帮我管理项目变更"               ← Skill 出现
2026: "帮我运营整个发布流程"            ← 多 Skill + 多 Agent
```

Skill 是这个演进中的**关键抽象**。它把 prompt engineering 的经验**产品化**了——你不需要知道怎么写 prompt，只需要知道"装上这个 Skill 就能做这件事"。

### 6.2 自定义 Skill 的潜力

对于团队来说，自定义 Skill 意味着：

- **规范自动化**：代码审查标准写成 Skill，每次审查都用同一套标准
- **经验编码化**：资深工程师的排查思路写成 Skill，新人直接用
- **流程可复制**：发布流程、on-call 流程、迁移流程——一次编写，全团队复用

最终形态：每个团队维护自己的 Skill 库，新人入职 = 加载一套 Skill，第一天就能按团队规范工作。

### 6.3 局限与挑战

- **泛化能力**：Skill 在定义好的场景中表现很好，遇到边界情况仍需人工介入
- **维护成本**：工具链变更、流程演进时，Skill 需要同步更新
- **安全风险**：Skill 越是拥有强大的工具权限，越需要严格的护栏设计
- **可组合性**：多个 Skill 如何在不冲突的情况下协同工作，是一个开放问题

---

Skill 不是一个复杂的概念——它就是**把人的经验翻译成 Agent 能执行的格式**。但它的出现标志着 AI 编程工具从"工具"进化为"工坊"：不只是给你一把锤子，而是把整个装配线配好了等你来用。

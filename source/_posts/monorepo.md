## 什么是 Monorepo

Monorepo（单一仓库）是一种源代码管理策略，将多个相关但独立的项目存放在**同一个版本控制仓库**中。与之相对的是 Polyrepo（多仓库），即每个项目独立维护一个仓库。

这里的关键区分不是"代码是否放在一起"，而是**原子化提交**和**统一版本控制**的能力 —— Monorepo 允许一次 commit 同时修改多个项目，而 Polyrepo 则需要跨仓库协调。

## Monorepo vs Polyrepo

| 维度         | Monorepo                     | Polyrepo                        |
|--------------|------------------------------|---------------------------------|
| 代码组织     | 所有项目在同一仓库           | 每个项目独立仓库               |
| 版本管理     | 统一版本，一次 commit 跨项目 | 独立版本，需跨仓库协调          |
| 依赖管理     | 共享依赖，统一升级           | 各项目独立管理依赖              |
| CI/CD        | 需增量构建支持               | 独立流水线，天然隔离            |
| 权限控制     | 粗粒度（仓库级）             | 细粒度（项目级）               |
| 代码复用     | 高，源码级共享               | 低，需通过包管理发布            |
| 工具支持     | 需要专门工具链               | 标准工具即可                    |

## 为什么要用 Monorepo

### 1. 原子化提交

一次需求变更可能同时涉及前端、后端和共享库。Monorepo 中所有这些改动可以放在一个 commit 里，回滚也只需一次 revert。

### 2. 代码复用成本低

共享代码无需走"发布 → 版本号 → 下游升级"流程，直接在仓库内引用源码即可。

### 3. 统一工具链与规范

ESLint、Prettier、TypeScript 配置只需维护一份，所有子项目继承生效。新人 on-board 时也无需了解每个仓库的不同配置。

### 4. 跨项目重构更安全

重构一个被多处引用的公共模块时，所有调用方都能在同一 PR 中看到编译结果和测试通过情况，不会出现"改了 A 仓库但 B 仓库不知道"的情况。

## Monorepo 的挑战

### 仓库体积膨胀

随着项目增多，clone 和 checkout 时间会显著增长。解决方案包括 shallow clone、部分检出（sparse checkout）以及 Git 的 filesystem monitor。

### 构建速度

全量构建不现实，需要增量构建 + 依赖图分析。主流工具对此有原生支持（见下文工具链部分）。

### 权限与治理

传统 Git 权限是仓库级别的，Monorepo 下所有人可见所有代码。需要通过 CODEOWNERS 文件做目录级权限控制。

### 工具复杂度

需要引入专门的 Monorepo 工具来管理 workspace、任务编排和依赖图。

## 主流工具链

| 工具          | 特点                                | 适用场景                      |
|---------------|-------------------------------------|-------------------------------|
| Turborepo     | 增量构建、远程缓存、任务并行编排    | 前端/全栈，JavaScript 生态    |
| Nx            | 依赖图可视化、代码生成器、插件丰富  | 企业级，多语言支持            |
| Lerna         | 版本管理和发布，常与 Nx 搭配        | npm 包发布管理               |
| Bazel         | Google 内部工具，支持多语言，可扩展 | 超大规模，多语言混合          |
| pnpm workspaces | 轻量，原生 workspace 支持         | 中小型 JavaScript Monorepo   |
| Rush          | 微软出品，强于依赖一致性和发布     | 大型 JS Monorepo，企业合规    |

## 常见的 Monorepo 目录结构

```
my-monorepo/
├── package.json              # 根 workspace 配置
├── pnpm-workspace.yaml       # pnpm workspace 声明
├── turbo.json                # Turborepo 任务编排
├── apps/                     # 应用程序
│   ├── web/                  # 前端应用
│   └── api/                  # 后端服务
├── packages/                 # 共享包
│   ├── shared-utils/         # 通用工具库
│   ├── ui-components/        # UI 组件库
│   └── eslint-config/        # 统一 ESLint 配置
└── tools/                    # 内部工具/脚本
    └── build-scripts/
```

## 什么时候该用 Monorepo

**适合：**

- 多个项目共享大量代码，且这些代码变更频繁
- 团队希望统一工具链和编码规范
- 跨项目重构经常发生
- 需要原子化跨项目提交

**不适合：**

- 项目之间几乎没有代码共享
- 团队分开管理，各自有不同的发布周期
- 仓库体积过大（数十 GB 级别），且无精力优化 CI
- 需要严格的代码访问权限隔离

## 小结

Monorepo 不是银弹，它是一种在"代码共享便利"和"工具链复杂度"之间的权衡。对于大多数中大型前端/全栈项目，结合 Turborepo 或 Nx 的 Monorepo 方案已经足够成熟，可以显著提升开发效率。

核心决策因素是：**你的项目之间共享了多少代码，以及共享代码的变更有多频繁**。如果答案是"很多"和"经常"，Monorepo 就值得认真考虑。
